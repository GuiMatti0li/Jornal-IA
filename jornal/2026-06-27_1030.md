**Google Acelera Gemini Nano em Pixel com Multi-Token Prediction Congelado**
A Google introduziu uma nova arquitetura que retrofita a predição multi-token (MTP) em modelos Gemini Nano v3 "congelados", acelerando a inferência on-device em dispositivos Pixel. Esta abordagem evita a necessidade de modelos "drafters" separados, otimizando o consumo de energia e memória.

A inovação permite que LLMs como Gemini Nano e Gemma funcionem de forma mais eficiente em telefones, superando gargalos de processamento token a token. O MTP head é anexado às camadas finais do modelo principal, alavancando os estados ocultos para prever sequências de tokens, mas sem precisar de treinamento conjunto ou um KV cache próprio. Isso resulta em um aumento significativo na velocidade e menor consumo de bateria para recursos como resumos de notificações e revisão de texto. **LLMs Ajudam Robôs a Entender Instruções Vagas e Focar em Detalhes Essenciais**
Pesquisadores do MIT CSAIL desenvolveram a abordagem "Masked Inverse Reinforcement Learning" (Masked IRL), que permite que LLMs ajudem robôs a interpretar comandos ambíguos e priorizar informações contextuais relevantes. O método reduz a necessidade de demonstrações extensivas em até cinco vezes.

O Masked IRL usa um LLM para expandir prompts pouco claros (ex: "fique perto" vira "fique perto da superfície da mesa") com base em dados de demonstração cinestética. Um segundo LLM avalia o ambiente e "mascara" detalhes irrelevantes, focando apenas nos críticos para o plano de movimento do robô. Em experimentos, robôs virtuais e reais manobraram objetos com até 15% mais precisão do que baselines comparáveis, demonstrando a capacidade de aprender preferências não explicitadas e evitar obstáculos com eficiência. **Overfitting em Avaliação de RAG: O Perigo da Reutilização de Testes**
Reutilizar o mesmo conjunto de testes para avaliar e refinar continuamente sistemas Retrieval Augmented Generation (RAG) leva ao overfitting, comprometendo a capacidade do modelo de generalizar para dados novos e não vistos. A avaliação adequada exige um conjunto de teste verdadeiramente independente.

O problema ocorre quando as equipes ajustam o RAG com base nos resultados de um conjunto de avaliação e, em seguida, reavaliam com o mesmo conjunto, transformando-o efetivamente em um conjunto de treinamento. Isso infla artificialmente as pontuações de desempenho. Para evitar o overfitting, é fundamental manter conjuntos de treinamento, validação e teste separados e usar o conjunto de teste apenas uma vez para uma avaliação imparcial do desempenho de generalização do sistema. [Towards Data Science](https://towardsdatascience.com/water-cooler-small-talk-ep-11-overfitting-in-rag-evaluation/)

**Exploit de pedit COW no Kernel Linux Permite Acesso Root via Binários Cachecados**
Uma falha no subsistema de controle de tráfego do kernel Linux, apelidada de "pedit COW" (CVE-2026-46331), permite que usuários locais não privilegiados obtenham acesso root, envenenando cópias em cache de binários.

A vulnerabilidade é uma escrita fora dos limites (out-of-bounds write) na ação de edição de pacotes (act_pedit) que corrompe a memória compartilhada do cache de páginas. O exploit não modifica o arquivo em disco, mas sim a cópia em memória de binários setuid root como `/bin/su`, injetando um payload malicioso que é executado com privilégios de root. Sistemas RHEL e Debian foram confirmados como vulneráveis com configurações padrão de user namespaces não privilegiados, que fornecem a capacidade `CAP_NET_ADMIN` necessária para o ataque. [The Hacker News](https://thehackernews.com/2026/06/new-linux-pedit-cow-exploit-enables-root-access-by-poisoning-cached-binaries/)

**Nova Falha DirtyClone no Kernel Linux Permite Root via Pacotes Clonados**
Uma vulnerabilidade recente no kernel Linux, a DirtyClone (CVE-2026-43503, CVSS 8.8), permite que usuários locais corrompam a memória mapeada para arquivos através de pacotes de rede clonados, levando à escalada de privilégios para root.

A falha decorre de duas funções auxiliares do kernel que, ao copiar pacotes de rede internamente, deixam de remover uma flag de segurança que indica que a memória do pacote é compartilhada com um arquivo em disco. Um atacante pode carregar um binário privilegiado na memória, como `/usr/bin/su`, e forçar o kernel a clonar pacotes que contêm essas páginas de memória. Ao passar por um túnel IPsec controlado pelo atacante, a etapa de descriptografia sobrescreve as verificações de login do binário, concedendo acesso root na próxima execução. A correção já foi incorporada no kernel mainline (v7.1-rc5) e versões LTS. [The Hacker News](https://thehackernews.com/2026/06/new-dirtyclone-linux-kernel-flaw-lets-local-users-gain-root-via-cloned-packets/)

**Falha Crítica no Amazon Q Developer Permitia Roubo de Credenciais Cloud**
Uma vulnerabilidade de alta gravidade (CVE-2026-12957, CVSS 8.5) na extensão Amazon Q Developer para Visual Studio Code permitia o roubo de credenciais de nuvem de desenvolvedores ao abrirem repositórios de código maliciosos.

A falha residia na forma como a extensão manipulava arquivos de configuração do Model Context Protocol (MCP), `.amazonq/mcp.json`, em workspaces, executando comandos sem a permissão explícita do usuário. Isso possibilitava que um repositório comprometido iniciasse servidores MCP locais, que herdavam todo o ambiente do desenvolvedor, incluindo chaves AWS, tokens CLI da nuvem, segredos de API e sockets SSH. A AWS corrigiu o problema na versão 1.65.0 do Language Servers para AWS (e versões posteriores recomendadas), que agora exige consentimento explícito para servidores MCP não confiáveis, além de ter corrigido uma falha relacionada (CVE-2026-12958) envolvendo links simbólicos. [The Hacker News](https://thehackernews.com/2026/06/amazon-q-developer-flaw-could-let-malicious-repos-run-code-via-mcp-configs/) / [SecurityWeek](https://www.securityweek.com/amazon-q-flaw-enabled-cloud-credential-theft-via-malicious-repositories/)
